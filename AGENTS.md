## 1. Project at a glance

The Nebius Cookbook is a public, GitHub-hosted collection of **production-grade recipes** for building AI agents on [Nebius AgentKit](https://nebius.com/services/token-factory). It launches at the Nebius customer conference on **June 9, 2026** (press and analysts in the room) together with a live keynote demo.

**Ship deadline: five cookbooks must be ready by June 4, 2026** (five working days before the keynote, to leave room for rehearsal, review, and last-minute fixes). Cookbooks #1 and #2 below are the first two; cookbooks #3, #4, and #5 are TBD (see section 16).

It is modeled on the best of:
- `developers.openai.com/cookbook` for pedagogical recipes
- `github.com/vercel/next.js/tree/canary/examples` for runnable starters

The two deliverables are:

1. **Cookbook recipes** — Python FastAPI apps, each a runnable production-shaped starter, each demonstrating one specific Nebius integration.
2. **Cookbook web app** — Next.js site that renders the recipes catalog with rich metadata, hosted on Clever Cloud.

The repo is a **monorepo** with both worlds living together: JavaScript (the app, shared packages, build scripts) and Python (the cookbook recipes themselves).

## 2. Hard constraints

These are non-negotiable. If a change would violate any of them, stop and ask.

- **Production-shaped, not toy-shaped.** Every recipe must look like code that could land in a real codebase. Type hints, error handling, observability hooks, sensible logging, healthchecks, Dockerfile. No `print()` debugging, no hardcoded credentials, no `except Exception: pass`.
- **Runnable in 5 minutes.** Any recipe that requires more than five minutes to clone, configure, and run on a fresh laptop fails the cookbook standard.
- **Each cookbook is autonomous.** Cloning a single cookbook directory must be sufficient to run it. No imports from sibling cookbooks. No shared Python packages. Duplication of skeleton code between cookbooks is accepted and expected.
- **Story before features.** Every recipe earns its place by demonstrating one concrete piece of the prototype → production journey for AI agents. If a feature is technically interesting but does not advance the story, cut it.
- **Boring infrastructure choices.** The stack is already opinionated. Everything else (HTTP framework, package manager, logging library, CI) defaults to the boringly conventional choice. No demonstrating personal taste in tooling.
- **No code that calls into networks during tests by default.** Use `respx` to mock Nebius and external APIs. Tests must pass with no network access.
- **Secrets never in code.** Always env vars, validated through Pydantic Settings at boot. `.env.example` lists every required key.

## 4. Package management

### 4.1 JavaScript / TypeScript: Bun

Bun manages the JS workspace at the root. Workspaces declared in `package.json` cover:
- `app/`
- `packages/*`

**Bun is NOT declared as a workspace manager for `cookbooks/`.** Those are Python projects, intentionally isolated from the JS world.

Use:
- `bun install` to install JS deps for the whole workspace
- `bun run <script>` to execute scripts declared in `package.json`
- Bun executes TypeScript directly. No transpile step. Files in `scripts/` are `.ts` and run via `bun run scripts/foo.ts`.

**Runtime:** Bun is used in dev and CI. **Production runtime on Clever Cloud is Node** (LTS). The `next build` output is a standard Node.js Next.js app. This means: nothing in `app/src/` should depend on Bun-specific APIs at runtime.

### 4.2 Python: uv per cookbook

Each cookbook is an independent uv project with its own `pyproject.toml` and `uv.lock`. **There is no root-level Python workspace.** This is intentional.

Use:
- `uv sync --frozen` inside a cookbook to install deps from the lockfile
- `uv run <command>` to execute commands in the cookbook's virtualenv
- `uv add <package>` to add a dep
- `uv lock` to refresh the lockfile

A contributor who only touches Python does not need Bun. A contributor who only touches the app does not need uv.

### 4.3 Version pins

- Node: `22.x` (in `.nvmrc`)
- Python: `3.12` (in `.python-version` at root and per cookbook)
- Bun: `>= 1.1` (declared in `package.json` via `"packageManager": "bun@1.1.x"`)

## 6. The root README composition

The root `README.md` is **NOT edited directly**. It is composed by `bun run build:readme` from sources in `README/`.

### 6.1 The pattern

`README/_config.json` declares an ordered list of sections. Each section is either:
- `"source": "human"` — points to a `.md` file in `README/`, included verbatim
- `"source": "generated"` — points to a generator in `scripts/generators/`, output included

The script writes the concatenated result to `README.md` at the repo root.

### 6.2 The `_config.json`

```json
{
  "$schema": "../schemas/readme-config.schema.json",
  "output": "../README.md",
  "separator": "\n\n",
  "header": {
    "comment": "This file is generated by `bun run build:readme`. Edit sources in README/ instead."
  },
  "sections": [
    { "id": "intro", "source": "human", "file": "01-intro.md" },
    {
      "id": "recipes",
      "source": "generated",
      "generator": "recipes-table",
      "options": {
        "cookbooksDir": "../cookbooks",
        "columns": ["order", "title", "stack", "difficulty", "readingTime"],
        "heading": "## Recipes"
      }
    },
    { "id": "stack", "source": "human", "file": "05-stack.md" },
    { "id": "getting-started", "source": "human", "file": "03-getting-started.md" },
    { "id": "contributing", "source": "human", "file": "04-contributing.md" },
    { "id": "license-credits", "source": "human", "file": "99-license-credits.md" }
  ]
}
```

### 6.3 The check mode

CI runs `bun run build:readme --check` which regenerates the README in memory and compares with the committed `README.md`. If they differ, CI fails. The contributor must run `bun run build:readme` locally and commit the result.

### 6.4 Cookbook READMEs are different

**Cookbook-level READMEs** (`cookbooks/*/README.md`) are hand-edited, single-file, NOT composed from sub-MD. The composition pattern applies only at the repo root level.


### 8.2 Dev dependencies

```toml
[dependency-groups]
dev = [
    "pytest >= 8.3",
    "pytest-asyncio >= 0.24",
    "respx >= 0.22",
    "ruff >= 0.7",
]
```

### 8.3 Ruff configuration

In each cookbook's `pyproject.toml`:

```toml
[tool.ruff]
line-length = 100
target-version = "py312"

[tool.ruff.lint]
select = ["E", "F", "W", "I", "B", "UP", "N", "ASYNC", "S", "RUF"]
ignore = ["S101"]  # asserts in tests

[tool.ruff.format]
quote-style = "double"
```

### 8.4 Nebius client wrapper pattern

`app/core/nebius_client.py` MUST:

- Use the OpenAI SDK with `base_url="https://api.studio.nebius.ai/v1/"` (verify the exact URL via Nebius docs at integration time)
- Read `NEBIUS_API_KEY` from Pydantic Settings, never directly from `os.environ`
- Configure explicit timeouts (`httpx.Timeout(connect=5, read=60, write=10, pool=5)`)
- Wrap calls in `tenacity` retry with exponential backoff for transient errors (429, 5xx)
- Return raw stream chunks for streaming endpoints — let the route handler format SSE events

### 8.5 Streaming SSE conventions

The agent endpoint emits SSE with **named events**, not raw text:

```
event: status
data: {"phase": "planning"}

event: token
data: {"text": "Snow"}

event: token
data: {"text": "flake"}

event: status
data: {"phase": "done"}

event: done
data: {}
```

Heartbeat every 15 seconds during long operations:
```
event: heartbeat
data: {}
```

On client disconnect (detected via `await request.is_disconnected()`), the route handler MUST cancel the upstream Nebius call.

### 8.6 Observability requirements

- **Logs are JSON.** structlog with a JSON renderer in production, console-friendly in dev (detected via `ENV` env var).
- **Every request gets a request ID.** Injected by middleware, returned in `X-Request-ID` response header, included in every log line.
- **Metrics on `/metrics`.** At minimum:
  - `http_requests_total{method, path, status}` counter
  - `http_request_duration_seconds{method, path}` histogram
  - `nebius_tokens_total{model, type=input|output}` counter
  - `nebius_request_duration_seconds{model}` histogram
- **No PII in logs.** If a prompt contains an email or phone, redact before logging.

### 8.7 Rate limiting

`slowapi` with sane defaults:
- 10 requests per minute per IP on `/agent/run`
- No limit on `/healthz`, `/readyz`, `/metrics`

### 8.8 CORS and security headers

In `main.py`:
- CORS allowlist read from settings, never `*` in production
- Security headers via middleware: `X-Content-Type-Options: nosniff`, `Strict-Transport-Security`, `Referrer-Policy: strict-origin-when-cross-origin`
- Request body size limit: 64 KB (enforced via Starlette)

### 8.9 Graceful shutdown

FastAPI lifespan handler:
- On startup: validate config, log version, register signal handlers
- On shutdown: close HTTP client, flush logs, drain in-flight requests with a 30-second timeout

## 10. Scripts

All scripts live in `scripts/` and are TypeScript executed by Bun.

### 10.1 `build-readme.ts`

Composes the root `README.md` from `README/` sources following `README/_config.json`. Supports `--check` for CI mode.

### 10.2 `generators/recipes-table.ts`

Generator invoked by `build-readme.ts`. Reads every `cookbooks/*/recipe.json`, sorts by `order`, produces a Markdown table.

### 10.3 `validate-recipes.ts`

Validates every `recipe.json` against `schemas/recipe.schema.json` using Ajv. Reports all errors at once (don't fail on first).

### 10.4 `build-recipes.ts`

For each cookbook:
- Reads `recipe.json` and `README.md`
- Compiles README to MDX using `@nebius-cookbook/mdx-pipeline`
- Writes to `app/src/content/recipes/<slug>.mdx`
- Builds `app/src/content/recipes-manifest.json`

### 10.5 `new-cookbook.ts`

Interactive prompt: title, slug, partner, difficulty. Copies `cookbooks/_template/` to `cookbooks/NN-<slug>/`, replaces placeholders, prints next-steps instructions.

### 10.6 `test-cookbooks.ts`

For each `cookbooks/*/`:
- `cd` into it
- Run `uv sync --frozen`
- Run `uv run pytest`
- Run `uv run ruff check`
- Aggregate exit codes; fail if any cookbook fails

### 10.7 `check-links.ts`

Walks all Markdown files in the repo, extracts links, verifies internal links resolve and external links return 2xx (with caching to avoid hammering external sites).

### 12.2 Cookbook #2 — "Real-World Data: Nebius + Tavily"

**Story.** The agent from cookbook #1 is locked in 2024 (its training cutoff). To brief a sales rep on tomorrow's call, you need to ground its answers in this week's news. Tavily + per-task model routing on Nebius makes this 10× cheaper than running everything through a frontier model.

**Story arc in the README:**

### 13.1 Python

- Type hints on all function signatures (return types included)
- Pydantic models for all I/O (no raw dicts in route handlers)
- Async everywhere in FastAPI handlers
- Imports in this order: stdlib, third-party, local (Ruff `isort` rule enforces this)
- One module per concern (don't put routes and models in the same file)
- No `print()`. Always `logger.info(...)` via structlog.
- No global state. Use FastAPI `Depends()` for client injection.

### 13.2 TypeScript

- Strict mode, no `any`
- Types derived from `recipe.schema.json` via `packages/recipe-schema/src/types.ts`
- Functional components only
- Tailwind classes inline, but extract to component when repeated 3+ times
- No CSS modules, no styled-components

### 13.3 Markdown

- ATX headings (`#` not underlines)
- Fenced code blocks with language tags always
- Reference-style links when a URL is reused, inline otherwise
- One sentence per line in source (helps git diff). Renders identically.

### 13.4 Commits

- Conventional Commits: `feat(cookbook-01): add streaming SSE`, `fix(app): handle empty recipe list`, etc.
- Scopes: `cookbook-NN`, `app`, `scripts`, `schema`, `readme`, `ci`, `docs`
- One logical change per commit

### 13.5 Branches and PRs

- `main` is always deployable
- Feature branches: `cookbook-NN/description`, `app/description`, `infra/description`
- PRs require all CI green before merge
- Squash merge by default

## 14. Workflow recipes

### 14.1 Bootstrap a new cookbook

```bash
bun run new
# Interactive prompts: title, slug, partner, difficulty
# Creates cookbooks/NN-<slug>/ from cookbooks/_template/
# Replaces placeholders
# Prints next steps
```

Then:
1. Edit `cookbooks/NN-<slug>/recipe.json`
2. Run `bun run validate` to confirm it's valid
3. Run `bun run build:readme` to refresh the root README
4. Implement the FastAPI app
5. Add tests
6. Add `Dockerfile`, `Makefile`, `.env.example`
7. Write the cookbook `README.md`
8. Open a PR

### 14.2 Work on a cookbook

```bash
cd cookbooks/01-first-agent-on-nebius
uv sync
cp .env.example .env
# Fill NEBIUS_API_KEY
make dev
# In another terminal:
curl -N -X POST http://localhost:8000/agent/run -d '{"prompt":"Hello"}'
```

### 14.3 Work on the app

```bash
bun install
bun run build:recipes
cd app
bun run dev
```

The app reads the manifest produced by `build:recipes`. Re-run it whenever a recipe changes.

### 14.4 Update the root README

Edit a sub-MD in `README/`, then:

```bash
bun run build:readme
git add README.md README/
git commit -m "docs(readme): update intro"
```

### 14.5 Deploy

The app deploys to Clever Cloud on push to `main`. Cookbooks deploy to Nebius Compute via `make deploy` inside each cookbook directory (the Makefile builds the Docker image and pushes to Nebius Container Registry, then triggers a deployment).

## 15. Things that look tempting but are wrong

- **A shared Python "base" package between cookbooks.** Tempting because it would reduce duplication. Forbidden because it would couple cookbooks together and break the "clone one folder and run it" promise.
- **A workspace `pyproject.toml` at the root.** Same reason.
- **Reading recipes at runtime in the Next.js app.** Tempting because it would simplify the build pipeline. Forbidden because it would impose a Node.js runtime requirement on the cookbook site and slow it down.
- **Editing `README.md` directly.** It will be overwritten on the next `bun run build:readme`. Edit `README/*.md` instead.
- **Using `os.getenv(...)` directly in Python code.** Always go through Pydantic Settings.
- **Adding LangChain to cookbook #1 or #2.** They are deliberately framework-free. LangChain is introduced in a later cookbook.
- **Adding mypy.** Ruff is enough for this scope. Avoiding mypy is intentional to reduce friction.
- **Switching to Granian, Litestar, or another non-Uvicorn ASGI server.** Uvicorn is the boring standard. Stick with it.
- **Using Bun as the runtime on Clever Cloud.** Build with Bun, run with Node. Configured in Clever Cloud dashboard.
- **Auto-generating cookbook READMEs.** Cookbook READMEs are hand-written. Only the root README is composed.
