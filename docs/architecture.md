# Architecture

This document explains how the Agent Blueprint Recipes monorepo is organized and why. For adding a cookbook, see [`adding-a-cookbook.md`](./adding-a-cookbook.md). For the contribution workflow, see [`../CONTRIBUTING.md`](../CONTRIBUTING.md).

## The two worlds

The repo houses two worlds that ship together but stay technically independent.

```
nebius-cookbook/
├── cookbooks/        ← Python world. uv per cookbook. No shared base package.
│   ├── 01-…/
│   ├── 02-…/
│   └── _template/
├── app/              ← JS/TS world. Next.js, Tailwind, deployed to Clever Cloud.
├── packages/         ← Shared TS — schema types, MDX pipeline. App-only.
├── scripts/          ← TS executed by Bun. Build, validate, bootstrap, test.
├── schemas/          ← Source of truth: recipe.json + readme-config schemas.
└── README/           ← Source files for the composed root README.
```

A contributor working on Python touches only `cookbooks/`. A contributor working on the site touches `app/`, `packages/`, and `scripts/`. The two are connected by:

1. `recipe.json` files (validated against `schemas/recipe.schema.json`)
2. Cookbook `README.md` files (compiled to MDX by `scripts/build-recipes.ts`)

## Why each cookbook is autonomous

Cloning a single cookbook directory must be enough to run it. This means:

- No imports from sibling cookbooks
- No shared Python "base" package
- No workspace-level `pyproject.toml`
- Skeleton code is **duplicated** between cookbooks, and that is intentional

The cost is that bug fixes need to be applied across cookbooks. The benefit is that a reader who is shown one cookbook can clone it, read it end-to-end, and never wonder "where does that import come from?" Pedagogical clarity wins here.

## Generation flow

```
schemas/recipe.schema.json
        │
        ├──► packages/recipe-schema/src/types.ts  (codegen)
        │           │
        │           └──► app/src/lib/recipes.ts   (typed manifest reader)
        │
        ├──► cookbooks/*/recipe.json              (validated by CI)
        │           │
        │           └──► scripts/build-recipes.ts ──► app/src/content/recipes-manifest.json
        │                                       └──► app/src/content/recipes/<slug>.mdx
        │
        └──► scripts/generators/recipes-table.ts ──► README.md (composed)
```

Every artifact downstream of `schemas/` is regenerated by a script. CI runs the generators with `--check` and fails if the working tree drifts. There is no "look at the file, edit it, commit" workflow for any generated file.

## Why Bun in dev/CI, Node in prod

Bun runs the TS scripts and the Next dev server without a transpile step, which keeps the inner loop fast. But Clever Cloud runs Node, and Next.js standalone output is a Node app — so production stays on Node 22 LTS. The lesson: nothing in `app/src/` should depend on Bun-specific runtime APIs.

## Why uv per cookbook

`uv` is fast enough that a contributor running a single cookbook never feels the cost of a per-cookbook venv. Each cookbook gets its own `pyproject.toml`, `uv.lock`, and `.venv`. The lockfile is committed. CI runs `uv sync --frozen` so the lockfile is authoritative.

## Why FastAPI

Async by default, Pydantic-native (so request/response models compose with `pydantic-settings` for free), and the auto-generated OpenAPI lets the cookbook site embed a live `/docs` view without extra work. Litestar and Granian are interesting but introduce friction for the audience we are writing for. Boring wins.

## Why structlog + JSON

A real service emits structured logs that a log pipeline (Loki, Datadog, etc.) can index without a parser. structlog with `JSONRenderer` in production and `ConsoleRenderer` in dev (toggled on `ENV`) is the smallest setup that ships both.

## The Nebius integration

The OpenAI Python SDK is used with `base_url` pointed at Nebius. This is deliberate: most readers have used the OpenAI SDK from a Python script. The "production-shape" lesson is everything else — config validation, retries, observability, deployment — not "learn a new SDK." The integration is one line.

## The deployment story

The Next.js app deploys to Clever Cloud on every push to `main` (build artifacts produced by Bun, runtime is Node). Each cookbook is deployable independently to Nebius Compute via its own `Dockerfile` and `make deploy`. The repo does not assume a single deployment target across cookbooks.
